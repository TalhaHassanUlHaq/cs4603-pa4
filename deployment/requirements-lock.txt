# Fully-pinned Python 3.12 / Linux dependency lock for the Model Serving container.
#
# WHY FULLY PINNED: the serving container runs a modern pip (26.x) that aborts with
# `resolution-too-deep` when the dependency graph is too deep to resolve. Pinning only
# the ~14 DIRECT deps was not enough -- pip backtracked through `databricks-connect`
# (pulled transitively via databricks-langchain -> unitycatalog-*[databricks]) across
# its 40+ releases and blew past its depth limit. Pinning the ENTIRE transitive closure
# to one version each gives pip a single candidate per package and zero backtracking.
#
# WHY PYTHON 3.12 (see deployment/deploy.py conda_env): databricks-connect 17.x (the
# only line that is both in-range for databricks-langchain AND numpy-2 compatible)
# requires Python ==3.12.*, while databricks-connect 16.x (the newest 3.13 build)
# caps numpy<2 and conflicts with mlflow/langchain (numpy>=2). On Python 3.13 there is
# NO databricks-connect that satisfies both, so the container is pinned to Python 3.12,
# which the Databricks ecosystem targets -- databricks-connect==17.0.10 + numpy 2.x
# then coexist. Verified: this exact set resolves for linux/py3.12 with zero backtracking.
#
# REGENERATE (after changing a direct dependency) with:
#   uv pip compile <14 direct reqs>.in --python-platform linux --python-version 3.12 \
#       --no-header -o deployment/requirements-lock.txt
